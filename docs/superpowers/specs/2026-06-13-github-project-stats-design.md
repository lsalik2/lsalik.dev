# Live GitHub Project Stats — Design

**Date:** 2026-06-13
**Status:** Approved (design), pending implementation
**Surfaces:** `projects/index.astro`, `projects/[...slug].astro`, `src/curl/render.ts` (via `middleware.ts`)

## Goal

Show each project's GitHub **stars** and **last-commit** recency on every
surface where projects appear — the project list and detail pages, in both the
browser and the `curl` terminal views — fulfilling the README's "live projects
github stats: stars, last commit, etc." Closed-source projects (no `repo`)
show no stats.

Displayed format: `★ 42 · updated 3d ago`.

## Decisions (locked)

- **Fetch timing:** build-time. The site is `output: 'server'` (per-request SSR
  on the edge); fetching in page frontmatter would call the GitHub API on every
  page view and hit the unauthenticated 60-req/hour limit. Instead a prebuild
  script fetches once and writes a static JSON the pages read.
- **Mechanism:** a standalone npm `prebuild` script (runs before `astro build`
  locally and on Vercel) writing a **committed** `src/data/github-stats.json`.
  Committing it makes dev/offline work and doubles as a last-known cache.
- **Fields:** `stars` (`stargazers_count`) and last commit (`pushed_at`). No
  language/forks/issues (language duplicates the existing `stack` field).
- **Stats key:** lowercased `owner/name`.

## Architecture

A build-time script fetches stars + last-commit per repo into a committed JSON
map. A small pure library parses and formats that data. The four render
surfaces read the map and emit a stats line. The network-touching script is
thin; all parsing/formatting/merging logic lives in the pure library and is
unit-tested.

```
prebuild script ── writes ──▶ src/data/github-stats.json
                                      │ static import
        ┌─────────────────────────────┼─────────────────────────────┐
   projects/index.astro      projects/[...slug].astro          middleware.ts
   (browser card line)       (browser meta block)        (terminal: render.ts)
        └──────────── all call statsForRepo() from src/lib/github-stats.ts ────┘
```

## Components / files

### `src/lib/github-stats.ts` (new — pure, fully tested)

```ts
export interface RepoStats { stars: number; pushedAt: string } // pushedAt = ISO 8601
export type StatsMap = Record<string, RepoStats>;               // key: lowercased "owner/name"

export function parseRepo(url: string): { owner: string; name: string } | null;
export function repoKey(owner: string, name: string): string;   // `${owner}/${name}`.toLowerCase()
export function statsForRepo(repoUrl: string | undefined, map: StatsMap): RepoStats | null;
export function repoApiToStats(json: { stargazers_count: number; pushed_at: string }): RepoStats;
export function formatStars(n: number): string;                 // 42 -> "42"; 1234 -> "1.2k"
export function relativeTime(iso: string, now: Date): string;   // "3d ago", "2mo ago"
export function formatStatsText(stats: RepoStats, now: Date): string; // "★ 42 · updated 3d ago"
```

- `parseRepo` handles `https://github.com/owner/name`, a trailing `/`, and a
  `.git` suffix. Returns `null` for non-GitHub or unparseable URLs.
- `relativeTime` buckets: `<60s` → `"just now"`; minutes `Xm ago`; hours
  `Xh ago`; days `Xd ago`; weeks `Xw ago` (7–27d); months `Xmo ago` (~30d);
  years `Xy ago`. `now` is injected so tests are deterministic.
- `formatStars`: `< 1000` → integer; `>= 1000` → one decimal + `k`
  (e.g. `1234` → `"1.2k"`, `1000` → `"1k"`).

### `src/data/github-stats.json` (new — committed)

The `StatsMap`. Generated by the script; committed so dev/offline renders and
so it serves as the last-known cache the script merges over.

### `scripts/fetch-github-stats.mjs` (new)

1. Read `src/content/projects/*.md`; parse frontmatter with `js-yaml` (already a
   dependency); collect each `repo` value.
2. For each repo: `parseRepo` → `GET https://api.github.com/repos/{owner}/{name}`
   with headers `Accept: application/vnd.github+json` and, if `process.env.GITHUB_TOKEN`
   is set, `Authorization: Bearer <token>` (raises the rate limit; optional —
   6 public repos fit comfortably in the unauthenticated 60/hour).
3. `repoApiToStats(response)` → `RepoStats`.
4. Merge results over the existing JSON: on any per-repo failure (network,
   non-200, parse), **keep the prior value** if present; otherwise omit that key.
5. Write `src/data/github-stats.json` (stable key order, 2-space indent).
6. Log a per-repo warning on failure and a one-line summary. **Always exit 0** —
   a GitHub outage must never fail the deploy.

The pure, testable pieces (`repoApiToStats`, the merge) live in
`github-stats.ts`; the script is the thin I/O shell around them.

### `package.json`

Add `"prebuild": "node scripts/fetch-github-stats.mjs"`. npm runs `prebuild`
automatically before `build`.

### Consumers

- `src/pages/projects/index.astro`: `import stats from '../../data/github-stats.json'`;
  per card, `const s = statsForRepo(project.data.repo, stats)`; if `s`, render a
  `.project-stats` line — `★ {formatStars(s.stars)} · updated {relativeTime(s.pushedAt, new Date())}`.
- `src/pages/projects/[...slug].astro`: same lookup; render a stats row in the
  meta block beneath the Repo row (omit when `s` is null).
- `src/curl/render.ts`: extend `ProjectSummary` and `ProjectPostFull` with
  `stats?: string` (a pre-formatted line). `renderProjectsIndex` appends the
  stats line to each project box; `renderProjectPost` appends it under the meta.
- `src/middleware.ts`: import the JSON; when building each summary/full object,
  set `stats = s ? formatStatsText(s, new Date()) : undefined` via `statsForRepo`.

## Data flow

`prebuild` → `github-stats.json` → static import in pages + middleware →
`statsForRepo()` → rendered line (browser markup via `formatStars` /
`relativeTime`; terminal string via `formatStatsText`).

## Error handling

- No `repo` / closed source → no stats line (all surfaces).
- Repo URL unparseable or key absent from the map → `statsForRepo` returns
  `null` → no line.
- Script fetch failure / rate-limit / private repo (404) → retain prior JSON
  value; omit only if never successfully fetched; build still succeeds.

## Testing

Unit tests in `tests/lib/github-stats.test.ts` for every pure helper:

- `parseRepo`: valid URL; trailing slash; `.git` suffix; non-GitHub URL → null;
  garbage → null; `undefined`-style empties.
- `repoKey`: lowercases and joins.
- `statsForRepo`: hit, miss (key absent), closed-source (`undefined` repo) → null.
- `repoApiToStats`: maps `stargazers_count`/`pushed_at` correctly.
- `formatStars`: `0`, `42`, `999`, `1000` → `"1k"`, `1234` → `"1.2k"`.
- `relativeTime`: one case per bucket (just-now, minutes, hours, days, weeks,
  months, years) with injected `now`.
- `formatStatsText`: composes `"★ N · updated Xd ago"`.

The network fetch itself is not unit-tested (matches how the OG integration's
I/O is handled); its pure transform and merge logic are factored into
`github-stats.ts` and tested there.

## Out of scope (noted, not built)

- Periodic auto-refresh via a scheduled Vercel deploy hook. Stats refresh on
  each push to `main` for now.
- Caching/runtime fetch (rejected in favor of build-time).
- Language / forks / open-issues fields.
